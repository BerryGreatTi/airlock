# .dev/

Development-time artifacts that support the coding process but are not part of the shipped product. If removing a file would break the running application, it does not belong here.

## Directory Structure

| Directory | Purpose | Examples |
|-----------|---------|---------|
| `scripts/` | Helper scripts, validators, one-off automation | Smoke test runners, data generators, migration helpers |
| `superpowers/` | Artifacts generated by superpowers skills | Brainstorming notes, review reports, intermediate outputs |
| `tmp/` | Ephemeral scratch files, simulation outputs | Experiment results, debug dumps, temporary fixtures |

## Rules

1. Production code must never import from `.dev/`. If something graduates to production-worthy, move it to the appropriate project directory.
2. `tmp/` is scratch space -- anything there can be deleted at any time without consequence.
3. Persistent documentation belongs in `docs/`, not here. Transient notes are fine, but anything referenced across sessions should be moved to `docs/`.
4. Keep scripts self-contained. Include a comment header explaining what the script does and when to use it.
